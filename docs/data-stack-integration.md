# Data Stack Integration

`tracking-plan.json` is designed to feed every layer of the modern data stack. Use `logline export` to convert it into formats each tool already understands.

## Segment Protocols

```bash
logline export --format segment --output segment-tracking-plan.json
```

Produces a Segment Protocols tracking plan with JSON Schema validation rules for each event. Upload via the Segment UI or API to enforce property contracts on all incoming events.

```json
{
  "display_name": "Workflow automation for teams",
  "rules": [
    {
      "name": "step_configured",
      "description": "User configured a step in their workflow",
      "rules": {
        "$schema": "http://json-schema.org/draft-07/schema#",
        "type": "object",
        "properties": {
          "properties": {
            "required": ["step_id", "workflow_id", "user_id"],
            "properties": {
              "step_id":     { "type": ["string"] },
              "workflow_id": { "type": ["string"] },
              "user_id":     { "type": ["string"] }
            }
          }
        }
      }
    }
  ]
}
```

## Amplitude Data Taxonomy

```bash
logline export --format amplitude --output amplitude-taxonomy.json
```

Produces an Amplitude event taxonomy you can import to enforce naming conventions and property types across your Amplitude workspace.

```json
{
  "events": [
    {
      "name": "step_configured",
      "description": "User configured a step in their workflow",
      "category": "core",
      "properties": [
        { "name": "step_id",     "type": "string",  "is_required": true },
        { "name": "workflow_id", "type": "string",  "is_required": true },
        { "name": "user_id",     "type": "string",  "is_required": true }
      ]
    }
  ]
}
```

## OpenTelemetry Semantic Conventions

```bash
logline export --format opentelemetry --output otel-conventions.yaml
```

Produces an OTel semantic convention YAML. Use this with the OTel Collector or to document your product events as custom semantic conventions alongside infrastructure spans.

```yaml
# Generated by Logline
groups:
  - id: step_configured
    type: event
    brief: "User configured a step in their workflow"
    stability: experimental
    attributes:
      - id: step_id
        type: string
        requirement_level: required
        brief: "Unique identifier of the step"
      - id: workflow_id
        type: string
        requirement_level: required
```

## GlassFlow

```bash
logline export --format glassflow --output glassflow-config.json
```

Produces a GlassFlow pipeline config with:
- **Filters** — pass/block rules per event name
- **Transforms** — validation rules for required properties
- **Schema** — full event schema for downstream consumers

```json
{
  "version": "1.0",
  "filters": [
    { "name": "filter_step_configured", "match": { "event": "step_configured" }, "action": "pass" }
  ],
  "transforms": [
    {
      "name": "validate_step_configured",
      "match": { "event": "step_configured" },
      "validate": { "required": ["step_id", "workflow_id", "user_id"] }
    }
  ],
  "schema": {
    "events": [
      {
        "name": "step_configured",
        "properties": [
          { "name": "step_id",     "type": "string", "required": true },
          { "name": "workflow_id", "type": "string", "required": true }
        ]
      }
    ]
  }
}
```

## dbt

The tracking plan's context graph maps directly onto dbt model relationships. Use the join paths to generate `ref()` chains:

```yaml
# models/step_configured.yml — generated from join path Step → Workflow → User
version: 2
models:
  - name: step_configured
    columns:
      - name: step_id
        tests: [not_null, relationships: {to: ref('steps'), field: id}]
      - name: workflow_id
        tests: [not_null, relationships: {to: ref('workflows'), field: id}]
      - name: user_id
        tests: [not_null, relationships: {to: ref('users'), field: id}]
```

The `context.joinPaths` field gives you the exact foreign key chain needed to build these relationships without manually inspecting your schema.

## Keeping exports in sync

Add to your CI pipeline to regenerate exports on every merge:

```yaml
# .github/workflows/tracking-plan.yml
- name: Update tracking plan
  run: |
    logline spec --fast
    logline export --format segment
    logline export --format glassflow
  env:
    OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
```

Or use `--fast` to skip the LLM and keep CI fast and free.
